# Tedxplore — Project Guide

Tedxplore is a specialized **no-code platform that generates premium TEDx-style event websites from structured event data**. It is NOT a website builder: users never touch layout, HTML, CSS, or design — they fill in structured content (speakers, sponsors, venue, FAQs, …) and the platform renders a professionally designed, animated, responsive single-page site.

Product philosophy: _"A premium event website generated automatically from structured event data."_

## Routing model (settled — don't re-derive)

- **One app, one domain.** Every event site is a _path_ on `tedxplore.com`: `tedxplore.com/tedx{slug}` (e.g. `/tedxmcgillu`). Not a subdomain, not a separate domain, not a separate deployment. Per-event custom domains are out of scope for V1.
- **The route is `src/app/[site]/`, not `src/app/tedx[slug]/`.** The App Router does not support partial dynamic segments — a folder named `tedx[slug]` matches nothing (verified empirically, returns 404). `[site]` receives the entire segment (`tedxmcgillu`) and strips the prefix via `parseTedxSegment` from `src/config/site.ts`.
- Build URLs with `tedxSitePath`/`tedxSiteUrl`, parse them with `parseTedxSegment` — never hardcode the `tedx` prefix at a call site.
- Static routes beat dynamic ones, so `/dashboard`, `/admin`, `/api/...` always take precedence over `[site]`. An event URL can therefore only collide with an app route that itself starts with `tedx`; the reserved-slug blocklist mainly prevents brand confusion (`/tedxplore`) and offensive URLs.

## Phase 1 decisions worth knowing

- **Slug minimum length is 2, not 3** (BR-1, `SLUG_MIN_LENGTH`) — changed during Phase 1.
- **`EventContent` uses `null`, never `undefined`, for absent values.** Snapshots round-trip through `JSON.stringify`, which drops `undefined` keys — an optional field would change shape between the preview and published paths.
- **BR-9 (one pending publish request per event) is enforced by the database** via `PublishRequest.pendingEventId`: it holds `eventId` while PENDING and NULL in every terminal state, so a plain unique index acts as a partial index. A raw `WHERE status = 'PENDING'` index would register as permanent Prisma schema drift. Only `publish-request-repository.ts` may write `status`, so the two can't diverge.
- **`EventContent` images carry no `alt` field.** Every V1 image slot has adjacent content that describes it better (speaker/sponsor/venue name); the hero background is decorative (`alt=""`). Templates derive alt text contextually.
- **Auth tables are hand-written to match Better Auth's core schema.** Phase 2 should run Better Auth's CLI generate and reconcile before building on them. `role` is a Prisma enum surfaced through Better Auth's `additionalFields`.

## Phase 2 decisions worth knowing

- **Next.js 16 renamed `middleware.ts` to `proxy.ts`** (exporting `proxy`, not `middleware`). The file is `src/proxy.ts`; the `edge` runtime is not supported there.
- **The proxy is not an authorization boundary.** It does an optimistic _cookie-presence_ check (`getSessionCookie`) and never validates a session. The real gate is `src/server/auth-guards.ts`, which every protected page, action, and handler must call. Never treat a path as protected merely because it appears in the proxy matcher.
- **Guards come in two flavours by caller:** `requireUser`/`requireAdmin` redirect (pages/layouts), `getAuthenticatedUser`/`getAdminUser` return a `Result` (Server Actions/route handlers, which must render an error rather than redirect mid-mutation). `getCurrentUser` is wrapped in React `cache`, so calling a guard repeatedly in one request is free.
- **Email verification is enforced by Better Auth, not by our code.** `requireEmailVerification: true` means no session is ever issued to an unverified account, so holding a session _is_ proof of verification (FR-3) — do not re-check `emailVerified` in guards.
- **`role` is `input: false`.** It cannot be set through sign-up or any client call. The only path to ADMIN is `pnpm exec tsx scripts/grant-admin.ts <email>` (`--revoke` to reverse). There is intentionally no admin-granting UI.
- **`returnTo` is attacker-controlled.** Always pass it through `sanitizeReturnTo`/`resolveReturnTo` (`src/lib/return-to.ts`) before navigating — an open redirect on a login page is a phishing primitive. Sanitize server-side in the page, then hand the clean value to client components.
- **Google OAuth and Resend degrade instead of failing.** Missing Google credentials hide the button and skip provider registration; a missing `RESEND_API_KEY` prints emails to the server console (which is how the flows were tested locally). `assertProductionIntegrations()` still refuses these modes at request time in production.
- **`src/config/env.ts` validates server env at module load.** CI's build step therefore supplies a placeholder `BETTER_AUTH_SECRET` — `next build` executes module scope without runtime secrets.
- **The hand-written auth tables were reconciled against `@better-auth/cli generate`** and match field-for-field; the sole deliberate deviation is `role` as the `UserRole` Prisma enum where the generator emits `String`. No migration was needed. (The CLI cannot load a config importing `server-only`; reconcile via a temporary standalone config file.)

- **Forms use React Hook Form + `zodResolver`** against the shared schemas in `src/lib/validation/`. API failures go to `form.setError("root", …)` and render in an `Alert`; field errors render in `FieldError`. Don't hand-roll `FormData` parsing.
- **`pnpm.overrides` pins a single `zod`.** `globals.css` imports `shadcn/tailwind.css`, so the `shadcn` package is a real build dependency — and it depends on zod 3.25, whose bundled `zod/v4` preview is a _different type identity_ from our zod 4. Without the override, `@hookform/resolvers` resolves the wrong copy and every `zodResolver` call fails to typecheck. Don't remove `shadcn` (the build needs it) and don't remove the override.
- **`SiteNav` takes `user` as a prop** rather than reading the session itself, so each layout controls where the session comes from and the signed-in state is in the first HTML. It is deliberately absent from `[site]` — public event sites render the organizer's template chrome, not ours.

## Phase 3 decisions worth knowing

- **The template registry shipped in Phase 3, not Phase 4.** FR-10 requires new events to be seeded with demo content, which needs a registry — so `src/templates/{registry,types}.ts` and `aurora/demo-content.ts` exist now. `TemplateDefinition` has no `Renderer` field yet (4.1 adds it); a placeholder component or an optional field would have been worse than its absence.
- **The demo seed is authored in _draft_ shape; `demoContent` is derived from it** by running the real serializer (`demoContent(template, now)` in `templates/types.ts`). It has two jobs — seed new drafts, render the homepage Live Preview — and maintaining an `EventContent` fixture alongside the seed would be two sources of truth. `demoSeed` is a _function of `now`_ because a hardcoded event date would eventually render the post-event state (FR-39) on brand-new drafts.
- **`prisma/seed.ts` seeds from the registry**, so local data matches what a real new event looks like.
- **Demo content deliberately references no images.** Cloudinary is Phase 5, and an image-free demo exercises the FR-38 hero fallback from day one.
- **`canManageEvent` lives in `src/server/services/authorization.ts`, not `auth-guards.ts`** (which re-exports it). It is a pure predicate; keeping it out of the guards module means services can apply it without dragging in `next/navigation` and `server-only` — which is also what makes the service layer runnable from a plain Node script for database verification.
- **A non-owner gets `NOT_FOUND`, never `FORBIDDEN`.** Confirming an event id exists is itself information. `loadManageable` in `event-service.ts` collapses the two cases; pages must not undo that by rendering a distinguishable error.
- **Server Actions return `Result`, never redirect.** A redirect thrown mid-mutation discards the field errors the form needs to render. Success navigation is the client's job (`router.push`). `src/lib/form-errors.ts` maps every `DomainError` to a field + message with an exhaustive switch — adding an error variant fails to compile until it has user-facing copy.
- **BR-5's slug lock is enforced twice, deliberately:** the service checks the state for a friendly error, and `updateEventSlug`'s write is scoped to `NEVER_PUBLISHED` so an event published between check and write still can't change its URL. Same for BR-3 — the pre-check is UX, the unique index is enforcement (`P2002` maps back to `SLUG_TAKEN` via `isUniqueConstraintError`).
- **Deletion mode is a shared pure rule** (`server/services/event-rules.ts`), imported by both the service and the confirmation dialog, so the dialog can't promise one outcome while the server does the other. Hard delete releases the slug; soft delete keeps it reserved forever.
- **The `shadcn` CLI can't run from the local install** — our zod-4 override breaks it (`s.deepPartial is not a function`). Use `pnpm dlx shadcn@latest add …`, which resolves its own zod 3, and decline the overwrite prompts for existing components.
- **Use `useWatch`, not `form.watch()`**, and never mutate a ref from a `register` callback — the React Compiler lint rules reject both. Where the create form needed "has the user taken over the display name?", it derives the answer from the values rather than remembering it.

## Phase 4 decisions worth knowing

- **Contrast must be computed in gamma space, and verified with Lighthouse.** A hand-rolled audit that composited alpha in _linear_ space passed everything; Chrome then failed `text-aurora-fog/60` at 3.68:1. CSS composites alpha in gamma space, so a light foreground at reduced opacity over a dark background is much darker than averaging suggests. Any new `/NN` opacity on text needs re-checking. The palette now carries three reds for measured reasons (`aurora-red` fills buttons, `aurora-ember` is the only red used for _text_, `aurora-red-deep` is the button hover) — never swap one for another to taste.
- **Aurora's fonts are the LCP budget.** `next/font` preloads every declared weight, and each weight of a non-variable Google font is a separate file. Archivo was declared at 600/700/800 while `aurora.css` only ever renders 700, putting ~80KB of unused glyphs on the critical path and holding LCP at 3.8s. Adding a heading weight means paying for a file; if several are ever needed, drop `weight` entirely and take the variable font. Note the root layout still preloads Geist Sans/Mono on public event sites. Phase 8 did **not** address this — it is logged against task 10.3, the final performance audit. Confirmed since: Aurora sets its own `font-family` for body and headings (`aurora.css`), so **Geist Sans is preloaded and never rendered** on an event site, while Geist Mono _is_ used by the countdown's `font-mono`. The fix is not a one-liner — a root layout cannot preload conditionally per route — so it is either `preload: false` or moving the Geist declarations down into the app/marketing/auth/admin layouts and giving Aurora its own mono. Measure before choosing: the public site scores 94 with this in place.
- **`TemplateDefinition` carries a `Poster` component, not a thumbnail path.** A screenshot is a copy of the template that nothing can keep honest — change the palette and the gallery advertises last month's design. The poster is drawn with the template's own tokens, so it cannot drift, ships no asset, and keeps "add Template 2" at one directory plus one registry entry.
- **Do not try to move the hero grain out of the LCP calculation.** Chrome attributes a pseudo-element's background image to its originating element, so `.aurora-grain::after` changes nothing — measured. (A run that appeared to show a large win was a stale server; always `fuser -k 3100/tcp` and rebuild between Lighthouse runs.) The grain was never the bottleneck anyway: removing it entirely left LCP at 3.8s.
- **Aurora animates with CSS, not Motion — `motion` has been uninstalled.** The first implementation used Motion's `whileInView`, which has to server-render its _pre-animation_ state: every section, including the hero's `<h1>`, shipped as `style="opacity:0"` and the page was blank until hydration. That delays LCP behind a JS download on a statically rendered site (NFR-1) and shows nothing at all with JS off. Reveals and parallax are now CSS scroll-driven animations (`animation-timeline: view()`), authored so the _visible_ state is the default and the animation is pure enhancement — browsers without support just show everything. (tech-stack.md now records this as decision 11; the stack table no longer names Motion.)
- **Above-the-fold content uses a time-based animation, not a scroll timeline** (`Reveal on="load"`). An element low in the first viewport has only partly completed its scroll range at scroll 0, so it would sit permanently half-faded. Same class of bug bit the nav anchors: the reveal range must end inside the `entry` phase, never at `cover N%`, or a section scrolled to via a nav link lands mid-fade.
- **The hero `<h1>` is deliberately not revealed at all** — it is the LCP element on every event site.
- **Aurora's tokens live in `src/templates/aurora/aurora.css`, imported from `globals.css`.** Tailwind v4 only reads `@theme` from its own entry graph, so a stylesheet imported by a component would be bundled but generate no utilities. Tokens are `aurora-` namespaced and base styles are scoped to the `.aurora` root, so the two design systems can't touch each other. Smooth scrolling uses `html:has(.aurora)` — `scroll-behavior` only works on the scrolling box.
- **`TemplateRenderProps` is `{ content, mode, now }`.** `now` is passed, not read: statically rendered pages would freeze `new Date()` at build time with nothing to reveal it. The countdown is the exception and ticks client-side, because the only correct clock is the visitor's.
- **Only three client components in the whole template**: the nav (scroll-spy), the countdown, and the speaker dialog. Each enhances markup that already works without it. The FAQ is native `<details>`/`<summary>` rather than a headless accordion — free disclosure semantics, and Ctrl+F opens the matching answer, which a JS accordion breaks.
- **lucide-react v1 dropped its brand icons** (`Instagram`, `Facebook`, `Linkedin`, `Youtube`, `Twitter` no longer exist). Social glyphs are hand-drawn paths in `components/social-icons.tsx`.
- **Templates use a plain `<img>`, not `next/image`.** Cloudinary already does format/quality/resize at the edge; `ImageRef` carries the dimensions that prevent layout shift. URL building is pure and lives in `src/lib/cloudinary-url.ts`, which returns `null` when `NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME` is unset so a missing config shows the FR-38 fallback rather than a broken image.
- **`next/font/google` is mocked for Vitest** (`src/test/next-font-mock.ts`, aliased in `vitest.config.ts`). It is a build-time transform, so without the mock every test that reaches the template registry fails at import.
- **Platform copy (About TED, About TEDx, disclaimer, default hero subtitle) lives in `src/config/platform-copy.ts`, not in the template.** It is a licensing obligation (A-2), so a second template must not be able to reword it — and keeping it out of frozen snapshots means corrections reach sites already published.
- **The template never states facts about the organizer's event.** No invented standfirsts ("eight talks", "a full day of ideas") — platform copy describes TED and TEDx only.

## Phase 5 decisions worth knowing

- **Autosave conflicts are a successful result, not an error.** `saveX` returns `{ updatedAt, conflicted }`; the write always lands (tech-stack decision 3's last-write-wins) and `conflicted` drives one page-level "edited somewhere else" banner. Refusing the write would be _first_-write-wins and would strand the person actively typing. `STALE_WRITE` therefore stays unused in `DomainError` — it's reserved for a future editor that can actually offer a merge.
- **`SaveStatus` is derived, not stored.** The hook stores only what the _save operation_ is doing (`idle/saving/saved/error`); `unsaved` is `form.formState.isDirty`. Keeping a separate `unsaved` state meant a `setState` inside an effect, which the React Compiler lint rules reject — and which let the indicator disagree with the form. Related: the lint rules also forbid mutating a ref during render, so the hook's "latest value" refs are synced in effects.
- **A validation failure leaves the status at `unsaved`, never `error`.** Nothing was attempted; the inline field error already says what's wrong, and "Save failed" would be a lie. Only transport-shaped failures schedule the single automatic retry — re-sending input the server already rejected would just overwrite the errors the user is reading.
- **The editor's wire format is deliberately not `EventContent`.** Text inputs submit `""`, the date arrives as wall time plus an IANA zone, and list rows carry database ids. `content.ts` schemas normalize `""`→`null` on the way in; `content/editor-defaults.ts` does `null`→`""` on the way out (a controlled input handed `null` silently goes uncontrolled and stops updating form state).
- **`draftToEditorDefaults` is not the serializer.** It drops nothing: a speaker with a blank name never reaches the public site (BR-13) but must stay visible in the editor to be fixable. Its schedule round-trip is pinned by a test — opening the editor and saving must not shift a published event's start time.
- **Wall time ↔ instant lives in `src/lib/datetime.ts`**, hand-rolled on `Intl` (no date library). A date with no timezone is _refused_ rather than defaulting to UTC: a silently-wrong countdown is worse than one more question. The two-pass offset resolution is DST-correct and unit-tested at the transition boundaries.
- **`prisma.event.update({ data: {} })` does not fire `@updatedAt`** — Prisma elides the statement entirely. Child-row writes bump the timestamp with an explicit `updatedAt: new Date()`; without it, adding a speaker left the dashboard claiming the event hadn't been touched. Found by a DB test asserting the timestamp moved, not by reasoning.
- **Repositories can't share one generic list helper via `tx[model]`** — Prisma's delegates are each generic in their own args type and a union of them has no callable signature. `content-repository.ts` gives each model concrete closures behind a narrow `ListOps<Fields>` interface instead, so every Prisma call stays fully checked and the orchestration is still written once.
- **Editor actions deliberately don't `revalidatePath`.** Autosave fires every ~1.5s of typing; every page showing this data calls a session guard and is therefore dynamic, and public sites render approved snapshots, not drafts. Phase 7's approval flow is what revalidates.
- **JSX drops a text chunk's leading space when the chunk contains an HTML entity and follows an expression container.** `up to {THEME_MAX_LENGTH} characters. …we&rsquo;ll…` renders as "up to 100characters". Removing _either_ the entity or the expression restores it (verified in the browser both ways). **Write the apostrophe as a literal `’`, not `&rsquo;`, in any line where an `{expr}` is followed by prose** — `{" "}` also fixes it but Prettier deletes the explicit space on the next format run, so it is not a stable fix. ESLint's `no-unescaped-entities` only flags ASCII quotes, so a literal `’` lints clean. One such site today (hero-section.tsx), commented.
- **List rows each own their form.** `useListEditor` owns only _which_ rows exist and in what order; each row runs its own `useForm` + `useAutosave`. Hoisting row values into a shared array would re-render every row on each keystroke.
- **Optimism is applied unevenly, deliberately.** Reorder and remove apply instantly and roll back on failure (both are reversible and the user is watching). Add waits for the server, because a row has no id until then and a temporary one would mean every later action — including the autosave 1.5s later — has to know whether the row is real yet.
- **Reordering is move-up/down buttons first, drag second.** The HTML5 drag API is unusable from a keyboard, and keyboard access is in the definition of done; both paths call the same idempotent full-order reorder. `@dnd-kit` would improve pointer ergonomics but is a large dependency for lists capped at 30 where the accessible path must exist anyway — the seam is `list-section.tsx` alone.
- **Never do side-effectful work inside a `setState` updater.** Updaters must be pure — React calls them twice in StrictMode — so a network call inside one fires twice per action. `useListEditor` reads `items` from state and depends on it instead. (Caught in review, before it shipped.)
- **A new FAQ row is seeded with placeholder text, not blanks**, because both halves are required on a row that exists (the serializer drops a question with no answer), so a blank new row would be invalid the instant it appeared.
- **Timezone and small enums use a native `<select>`** (`components/ui/native-select.tsx`), not shadcn's listbox: ~400 IANA zones get platform search, virtualization, and mobile pickers for free, and `register` works without a `Controller` per field.

- **Never write `DateTime` columns with raw `pg` — use Prisma.** Prisma maps `DateTime` to `timestamp(3)` _without_ time zone and treats the stored wall time as UTC; node-postgres treats the same column as **local** time. A one-off repair script written with `pg` therefore stored 09:00 where Prisma expected 14:00, and the demo event's start time silently moved five hours (caught by eye in the draft preview, not by any test). Verification scripts that only read are fine; anything that writes must go through Prisma or the service layer.

### Phase 5.4 (Cloudinary) decisions

- **No Cloudinary SDK.** Three HTTP calls and a SHA-1; the package's dependency tree buys nothing. Same trade as `lib/datetime.ts`.
- **The upload never passes through our server.** The server signs a constrained ticket and the browser posts straight to Cloudinary, so a 10 MB body never occupies a function. Verified: `folder`, `public_id`, and `overwrite` are all covered by the signature, and swapping any of them gets a 401 from Cloudinary — tested all three.
- **`attachImage` is the trust boundary, not the upload.** The client's report is treated as a hint about _which_ asset to look up and nothing more: the service checks the id is inside `tedxplore/events/{eventId}/`, then re-reads format, dimensions, and byte size from the Admin API and writes `MediaAsset` from _that_. Believing the client would let it lie about dimensions templates use to reserve layout space, or claim another event's asset. Both holes are tested.
- **The public id is a server-chosen UUID.** Not the file name (user input that would land in a public URL) and not slot-derived (which would collide on replace, forcing either an overwrite that mutates an asset a live snapshot still points at, or a suffix guessing game).
- **Removing an image clears the FK and nothing else.** The `MediaAsset` row and the Cloudinary asset both stay: snapshots are immutable (invariant 3) and a published site may still be rendering that asset. Task 10.4's orphan sweep is what reclaims them — which is also why every inbound FK is `SetNull`.
- **SVG is rejected, including for sponsor logos.** FR-21 permits this ("otherwise rejected"). An SVG can carry script, and every rendition is served through a transform that rasterizes it anyway (FR-22) — so accepting one would buy nothing but the risk. `ACCEPTED_SPONSOR_LOGO_CONTENT_TYPES` stays in `limits.ts` for whenever sanitization is actually built.
- **Format and size are enforced against Cloudinary's reading of the bytes**, never the browser's `File.type`/`File.size`. The client-side checks exist only so the user finds out before pushing 10 MB up a phone connection.
- **Signing lives in `lib/cloudinary-signature.ts`, not the adapter** — pure, secret-as-argument, so it is unit-testable without an account or a network. Same split as `lib/cloudinary-url.ts`. A silent change here means every upload 401s, so the tests pin the algorithm rather than a stored digest.
- **`ImageField` is deliberately outside the autosave forms.** A file picker has no "still typing" state: the work should start on selection and the result is one atomic change, so the control owns its own request lifecycle instead of being a field inside `useAutosave`.
- **`cloudinaryUrl` percent-encodes the public id**, so folder slashes arrive as `%2F`. Cloudinary serves both forms (verified, 200 either way); noted only because the two spellings are separate CDN cache keys.

## Phase 6 decisions worth knowing

- **Preview tokens are stored in plaintext, deliberately** — a departure from how reset tokens are treated. This one gates a read-only view of the owner's own draft, and the owner must be able to re-read the link to share it again; hashing would mean showing it exactly once, which is wrong for a link whose job is being pasted into a group chat.
- **"Create" and "regenerate" are one operation.** `issuePreviewLink` revokes the outstanding token in the same transaction, so there is never a window with two live links and no client can guess wrong about which verb applies.
- **Every failure to resolve a token is the same `NOT_FOUND`** — malformed, unknown, revoked, and belonging to a soft-deleted event. Distinguishing them would confirm an event exists and that someone thought better of sharing it. The shape check (`isPreviewTokenShaped`) runs first so probing the namespace costs no query, which is also what will let Phase 9.4's rate limiter measure real guesses.
- **The dead link is a segment-scoped `not-found.tsx`, not a component the page returns.** The platform 404 gets the message wrong ("you typed something wrong" is untrue and actionless), so the branded copy is the point. It was _also_ chosen because a boundary yields a real 404 where a returned component is a 200 saying "gone" in prose — **but Phase 8.0 lost that half; see below.**
- **`/preview/[token]` no longer declares `dynamic = "force-dynamic"` — Cache Components forbids it, and the guarantee survives as a default.** Uncached is now the default, so the token lookup is still per-request and FR-26 still holds. The rule this leaves behind: **never wrap that lookup in `use cache`**, however tempting the per-request query looks. Re-verified end to end (`scripts/verify-8-0.ts`).
- **A revoked preview link now returns 200, not 404 — deliberately, and it cannot be cheaply undone.** Once a Suspense fallback renders (`preview/loading.tsx`), the server has committed to `200 OK` and cannot revise the status; `notFound()` still renders the branded page and Next injects `<meta name="robots" content="noindex">` in the status's place. Recovering the 404 would mean resolving the token _before_ any boundary, which Cache Components rejects as an uncached read (`connection()` and dropping the boundary were both tried — both fail the build with `blocking-route`); the only remaining route is a database lookup in the proxy, which contradicts the proxy's whole design. **What FR-26 protects — the draft ceasing to be served on the very next request — holds, and FR-27's noindex holds twice over.** The lost status is read by link checkers, not people. `scripts/verify-8-0.ts` asserts the 200, so a future Next.js that restores the 404 fails loudly instead of leaving the docs wrong.
- **FR-27 is enforced twice:** `robots` metadata on the page and `X-Robots-Tag: noindex, nofollow` from `next.config.ts` on the whole `/preview` prefix. The header reaches crawlers that never parse HTML. `next.config.ts` imports `PREVIEW_PATH_PREFIX` from `config/routes.ts`, which is why that module imports `./site` **relatively** — the config is loaded outside the app's module graph, where `@/` does not resolve.
- **`DraftSite` is shared by both preview routes.** The promise of a preview is that it is not an approximation; two call sites each assembling "serializer → template → preview mode" is two places for that to stop being true. Authorization stays in the routes — the component renders and decides nothing.
- **These two actions _do_ `revalidatePath`, unlike the rest of the editor's.** The rule was never "actions don't revalidate" but "autosave fires every 1.5s"; issuing or revoking a link is deliberate and occasional, and the editor page server-renders the current link. Verified in the browser that this does not close the open dialog.
- **Running a service from a plain Node script needs two module shims**, not one: `next/font/google` (a build-time transform, in the registry's import graph via `event-service`) and `server-only` (which throws outside an RSC graph — resolve it to the package's own `empty.js`). Do **not** reach for `--conditions=react-server` to fix the second: it also swaps React for a build with no `createContext`, and lucide-react dies on import.

## Phase 7 decisions worth knowing

- **`Event.suspendedFromStatus` is a new column, and it had to be.** Restoring a suspension has to put the event back where it was, and `liveSnapshotId` cannot tell you: it is deliberately retained across an unpublish (BR-8a), so a suspended site that was live and one the owner had already taken offline look identical. Without the column, restore either forces a private site public or leaves a wrongly-suspended site dark. `restoredStatus` also falls back to `UNPUBLISHED` for rows suspended before it existed.
- **Suspension applies to `UNPUBLISHED` events too, not just live ones.** An unpublished event can be put back live by its owner instantly and without review (`canRepublish`), so if suspension only covered live sites an owner could unpublish the moment a report landed and republish the moment the admin looked away. Suspending is the only action that actually stops that.
- **A suspended owner cannot submit for review.** Otherwise a resubmission approved in the ordinary queue would silently undo another admin's suspension — BR-10 makes lifting one an admin-only act, and the queue is not that act.
- **`INCOMPLETE_CONTENT` carries `CompletenessIssue[]`, not `string[]`.** The one place `DomainError` reaches into a domain module, and worth it: FR-30 asks for a clear list of what's missing, and "clear" means each item links to the editor section that fixes it. Flattening to names at the error boundary would force the UI to map messages back to sections by guesswork. `SECTION_ANCHORS` in `publish-panel.tsx` maps the completeness vocabulary to the editor's anchor ids (only `basics` → `hero` actually differs), so neither module has to bend to the other's naming.
- **Cache invalidation lives in the actions, never the services.** Two reasons, and the first is the hard one: `next/cache` only works inside a request context, and the services are deliberately runnable from a plain Node script — which is how every phase has been verified against the real database. Services return the affected slug; the action calls `revalidateSite(slug)`.
- **`updateTag`, not `revalidateTag`.** Next 16 splits them: `revalidateTag` still serves the stale copy on the next request, `updateTag` makes that request wait for fresh data. For FR-44 that difference _is_ the requirement — one more visitor served a suspended page from cache is exactly what the admin pressed the button to prevent. The cost is that `updateTag` is Server-Action-only; every caller is one. The tag itself is `siteCacheTag(slug)` in `config/site.ts`, named once because Phase 8's `[site]` route has to attach the identical string and a typo would fail silently.
- **Publishing emails are fire-and-forget; auth's are not.** A verification email that fails must fail the sign-up, because the alternative is an account nobody can activate. By the time a publishing email sends, the state change has already committed — propagating a Resend outage would tell an admin "approval failed" for an approval that succeeded, and the retry would hit a request that is no longer `PENDING`. `notify()` logs and moves on.
- **The review page renders the snapshot inline, not in an iframe.** An iframe needs a _route_ to point at, and that route would serve unapproved content to whoever guessed its URL. Rendering inline keeps the snapshot inside the authenticated request that already loaded it — possible only because Phase 4 namespaced Aurora's tokens and scoped its base styles to `.aurora`.
- **The reviewer sees the snapshot, never a re-serialized draft.** FR-31 lets the owner keep editing while a request is pending, so the draft has usually moved on. Approving something other than what was reviewed is the failure that screen exists to prevent. A snapshot that no longer parses `eventContentSchema` throws rather than rendering partially — it means `EventContent` changed without the Phase 8 upgrader being taught about it.
- **Approve is one click, reject opens a dialog; suspend opens a dialog, restore is one click.** The dialog appears where there is something to _write_, not merely something to confirm. A confirmation step on approve — after the reviewer has just scrolled the whole site — only trains them to dismiss dialogs.
- **A rejection reason has a 20-character minimum (`REJECTION_REASON_MIN_LENGTH`).** It is the entire body of the organizer's email and the only thing telling them what to fix; "no" clears a non-empty check and is a dead end. A suspension reason is deliberately optional — a suspension may be urgent, and requiring an essay first is the wrong trade.
- **Rejection never touches `publicationStatus`.** A live site whose resubmission is rejected stays live on its last approved snapshot (BR-8a); taking it down would punish an organizer for trying to improve it. Correspondingly, `rejectRequestAction` is the one admin action that does not call `revalidateSite`.
- **Cancelling a submission that is no longer pending is a refusal, not a silent success** — the opposite of `revokePreviewLink`, deliberately. Revoking a link twice reaches the requested state either way; cancelling a just-decided submission usually means it was _approved seconds ago_, and reporting "done" would leave the owner believing they stopped a review that in fact published their site.
- **Services re-check the admin role even though `/admin` has a layout guard.** A layout guard protects a _route_; the day one of these services is called from a route handler or a script, the route is not what is protecting it. Same reasoning that makes the proxy explicitly not an authorization boundary.
- **The JSX entity/whitespace bug from Phase 5 bit again**, in the publish panel's "…publish it at {url} once it's approved" line — it rendered as "tedxavelorneonce". Caught in the browser, not by any test. The rule stands: in any line where an `{expr}` is followed by prose, write the apostrophe as a literal `’`, never `&rsquo;`.
- **Verification scripts need `tsconfig.script.json`.** `tsx` resolves `server-only` and `next/font/google` through CJS, so an ESM loader hook never fires — the working shim is tsconfig `paths` mapping `server-only` to its own `empty.js` and `next/font/google` to a stub. Run with `pnpm exec tsx --tsconfig tsconfig.script.json <script>`.

## Phase 8 decisions worth knowing

- **Enabling `cacheComponents` was never a two-file change, and the plan's estimate was wrong about that.** Task 8.0 named the two route segment configs; the flag actually failed the build on **15 routes**, because under Cache Components any uncached `await` above a `<Suspense>` boundary is a fatal `blocking-route` error. Three distinct causes: session reads (`getCurrentUser` → `headers()`), `await searchParams` on the auth pages, and `await params` + a database read on `/preview/[token]`.
- **A `loading.tsx` does not cover its own layout's `await`.** It renders _inside_ that layout, so it wraps the page and everything below, never the layout itself. This is why `(app)` and `admin` needed their layouts restructured while `(auth)` and `preview` needed nothing but the boundary file — their layouts are synchronous.
- **The layout guards moved into streamed components, and that is safe only because every page guards itself.** `AppLayout`/`AdminLayout` used to `await requireUser()`/`requireAdmin()` at the top and were the stated single gate. They now stream that call inside the nav. **Verified at the time of the change that all four `(app)` pages and all four `admin` pages call their own guard**, and the admin _services_ re-check the role independently — so the protection is three-deep (proxy → page guard → service guard), not one-deep. A new page added to either group without its own guard silently breaks this; it is the one thing to check in review.
- **`AdminNavLink` is a client component calling `usePathname()`, which is runtime data on a dynamic route.** That is why the whole admin section nav streams rather than just the pending-count badge — on `/admin/review/[requestId]` the _links_ were blocking before the count was. A nested boundary around the badge alone would not have covered them.
- **The nav is split into `SiteNavShell` + `SiteNavActions` so the bar can prerender and only its right-hand side streams.** Kept whole, one session read made the entire header — and therefore every layout's first paint — request-dependent.
- **The public homepage is now static, with the Edit button as its only streamed island.** Reading the session at the top of the page (which 4.9 accepted) made the whole marketing surface dynamic to decide one `href` per card. `TemplateCard` takes an `editAction` slot; `TemplateEditButton` reads the session itself. `templateEditHref` is untouched — FR-51 behavior is byte-identical, verified: signed out still yields `/login?returnTo=%2Fdashboard%2Fevents%2Fnew%3Ftemplate%3Daurora`.
- **Streaming costs you HTTP status codes and server-side redirects.** Once a fallback renders, the status is committed to 200; a mid-stream `notFound()` becomes 200 + injected `noindex`, and a mid-stream `redirect()` becomes a client-side redirect. This is what cost `/preview/[token]` its 404 (see the Phase 6 notes). Any future route where the _status_ is load-bearing must resolve that condition before its first boundary — or reject in the proxy, which is the only layer that still runs before rendering.
- **`notFound()` must stay outside a `use cache` scope.** It works by throwing, and a throw is not a value a cache entry can hold — hence the template Live Preview checks the registry in the page and re-looks-it-up inside the cached component. Passing the `TemplateDefinition` in was not an option either: cached-function arguments form the cache key and must be serializable, and it carries React components.
- **`cacheLife("hours")` is not identical to `revalidate = 3600`.** Same 1h revalidate, but `expire` drops from 1 year to 1 day. For the demo preview that is fine-to-better (a fresher countdown); worth knowing before reusing the profile somewhere the difference matters.
- **Verification scripts that hit a running server belong in `scripts/`, not the scratchpad** — `tsconfig.script.json`'s `paths` only resolve for files inside the repo.
- **`[site]` declares `generateStaticParams` for the HTTP status code, not for the prerendering.** Under Cache Components a dynamic segment without it makes `params` runtime data, which must be read below a `<Suspense>` boundary — and once a fallback renders the response has committed to 200. That is exactly what cost `/preview/[token]` its 404. With `generateStaticParams`, `params` is build-time data, so the slug resolves before any boundary and `notFound()` still sets a real 404. On a public SEO surface that status is load-bearing. The prerendered list being stale is harmless: an unlisted slug renders on first request and is written to disk from then on, which is why there is deliberately **no `dynamicParams = false`** — that would 404 every newly approved event until the next deploy.
- **`generateStaticParams` must return at least one param under Cache Components** (`empty-generate-static-params`), and it runs during `next build`, where there may be no database at all — CI builds with only a placeholder auth secret. So the list is fetched inside a `try`/`catch` that falls back to none, and an empty result yields one placeholder param built from the reserved slug `plore` (BR-4 guarantees no event can hold it). The documented cost is that a build with nothing published never exercises the real render path.
- **Next.js refuses `new Date()` in a prerendered Server Component** (`next-prerender-current-time`) — reading the clock while producing a page served to many people at other times is a bug in the general case. `now` is therefore captured _inside_ the `use cache` scope and returned alongside the content, where it has a precise meaning: the moment that cache entry was produced. That is exactly what Aurora's footer already documents its copyright year as being, and `cacheLife("days")` bounds how far it can lag.
- **`loadLiveSite` returns `null`, not a `Result` — the one service that does.** Every other service returns a discriminated union because callers must tell "not found" from "not yours" from "not in that state". FR-42 requires the opposite: suspended, unpublished, deleted, never-published, and nonexistent must be _indistinguishable_. A `Result` would offer an error variant to branch on, and the first branch anyone added would leak. The repository's `where` clause does the collapsing so it cannot be undone downstream — and the verification script asserts the two responses are byte-identical once the echoed URL is masked out.
- **`[site]` is the 404 for every unmatched single-segment path**, not just event URLs — `/typo` renders `[site]/not-found.tsx` rather than the root one. Its copy leads with the general case and mentions publication second for that reason.
- **There is deliberately no route that calls `revalidateSite` without authorization.** `updateTag` is Server-Action-only, so verifying the invalidation seam needs one; `scripts/verify-8-1.ts` documents a throwaway page to add, run, and delete, and skips that section loudly when it is absent. A permanent one would be a free cache-invalidation endpoint for anyone who found it.
- **`revalidateSite` updates two tags, not one.** The per-site tag and `SITEMAP_CACHE_TAG`, because every caller is an action that moves a site into or out of the published set — which changes the sitemap's _contents_, not just one page. Doing it inside `revalidateSite` rather than at the four call sites means a fifth publishing action cannot forget it.
- **A published site's `<title>` uses `title.absolute`**, escaping the root layout's `%s · Tedxplore` template. The site belongs to its organizers; a tab or search result reading "TEDxMcGill University · Tedxplore" puts the platform's name in front of theirs on their own page. The platform is credited in the footer, which is where the licensing obligation actually places it.
- **The sitemap omits non-live sites rather than disallowing them**, and `lastModified` is the live snapshot's `createdAt`, not `Event.updatedAt` — the latter moves on every autosave of a draft that may never be submitted. The template Live Preview is `Disallow`ed in `robots.ts` because it is a complete-looking event site for an event that does not exist, and indexed it would compete with real organizers' sites.
- **Next.js owns the metadata of a `notFound()` render.** `generateMetadata` returning a title for a non-live slug is dead code — Next supplies the site title and injects `<meta name="robots" content="noindex">` itself (verified over HTTP, on a route that still returns a real 404).
- **Snapshot migrations are typed against plain JSON, never `EventContent`.** The typed shape describes only today's version, so a v1→v2 migration written against it would compile now and silently mean something else once v3 lands. Plain JSON in, plain JSON out, one schema parse at the end. Upgrading runs on every read and never rewrites a row — invariant 3 makes a backfill migration exactly the wrong move, since a snapshot is the record of what an admin approved.
- **A snapshot from a newer deployment is refused, not partially parsed.** Reachable during a rolling deploy; parsing a v2 document with a v1 schema would drop whatever is new and look like content vanishing from a live site rather than like a deploy in progress.

## Phase 9 decisions worth knowing

- **The report affordance reaches the template through a `reportSlot` on `TemplateRenderProps`, not through `EventContent`.** FR-45 wants the link in the site's own footer, but a report must name the event and `EventContent` carries no id or slug (C-1). The route builds the node (`<ReportDialog slug={slug} />`) and the template positions it; the template still can't reach the database. Only the public site passes one — the homepage demo and both preview routes pass nothing, so a template must treat the slot as absent-by-default.
- **The report validation schema is split: `reportFormSchema` (transform-free, what the browser binds) and `reportSubmissionSchema` (what the server parses).** Partly a React Hook Form constraint (a schema whose input/output types differ can't be bound), but mainly a security requirement: **the honeypot is validated only on the server.** A `max(0)` on it client-side returns a 400 where a real report returns 202/202-equivalent, and that difference is the exact signal a bot needs to learn which field to leave alone. `scripts/verify-9-2.ts` caught this — I had written the `max(0)` on the submission schema and it made the trap a 400.
- **The report endpoint returns the same "accepted" for a stored report, a caught honeypot, and a slug that isn't a live site.** A distinguishable response would teach a bot its honeypot was caught, and would make the endpoint a slug oracle — conceding on the write path what FR-42 conceals on the read path. The rate limit is the _only_ honest refusal (429), because the reporter needs to know their report didn't land.
- **IPs are HMAC'd with `BETTER_AUTH_SECRET`, never stored raw or plainly hashed.** IPv4 is ~4 billion values, so an unkeyed SHA-256 is reversible by anyone who gets the database. Consequence: rotating that secret orphans existing report hashes and resets rate-limit windows — acceptable, since none of it is a durable record. `lib/client-ip.ts` reads the _first_ `x-forwarded-for` entry (the client; later entries are our own hops) — reading the last would rate-limit our own edge, one bucket for all traffic.
- **Closing a report is a guarded `updateMany WHERE status='OPEN'`, and resolve/dismiss stay distinct.** The guard stops two admins overwriting each other (a losing write is `INVALID_STATE`, not a silent reopen). Resolve vs. dismiss record opposite conclusions about the site — acted-on vs. nothing-to-do — which is what later tells a repeat offender from a repeatedly-reported innocent. **Suspension is a link to the event page, never a button on the report**: taking a site offline should require the whole event in view, not one stranger's paragraph, and closing a report deliberately does not `revalidateSite`.
- **Two rate limiters, two stores, on purpose.** Our `RateLimiter` adapter (`RateLimitWindow`, fixed-window) owns report submission and preview-token guessing; Better Auth's own DB-backed limiter owns the auth surface. Better Auth already knows every auth path, so tightening the brute-forceable ones is a `customRules` table rather than hand-parsing the `/api/auth` catch-all — chosen over forcing our adapter in. `storage: "database"` (not the in-memory default) because Fluid Compute resets memory on cold starts and splits it across instances.
- **The preview-guessing limit counts failures only, via a `peek` that never writes.** A valid token resolves and returns before any `consume`, so legitimate viewers never spend the budget; malformed tokens cost no query and so cost no write; only a shaped-but-invalid guess consumes. Once over budget, `peek` (a single indexed read) turns the guess away before the lookup — that's the DB protection. A rate-limited guess is still an indistinguishable `NOT_FOUND`, unlike the report 429: a guesser is owed no signal.
- **Better Auth rate limiting is `enabled: true`, not left to its production-only default.** So the same limiting code path runs in dev and can be verified locally. The limits are loose enough (5 sign-ins/min) not to bite normal development. A real sign-in still returns 401, not 500 — the config does not break auth.

## Architectural invariants (never violate)

1. **Content vs. presentation separation.** The DB schema contains zero template-specific fields. Templates are pure renderers: `(content: EventContent, mode) → React tree`; they never touch the database. The editor is template-independent.
2. **`EventContent` is the contract** (`src/content/event-content.ts`, Zod, versioned via `schemaVersion`). Drafts (relational tables) serialize into it; snapshots freeze it as JSON; templates consume only it. Any change to it must update the Zod schema, serializer, template, and snapshot upgrader together, in one change.
3. **Snapshots are immutable.** Every submission creates a new snapshot; snapshots are never mutated; all are retained for audit/restoration. The public site renders **only** the latest approved snapshot — never draft tables.
4. **Layering:** routes/actions are thin (auth → Zod validate → service). Services are pure TS domain logic returning discriminated-union results. Only `src/server/repositories/` imports `@prisma/client`. External services (Cloudinary, Resend, rate limiting) live behind adapters in `src/server/adapters/`.
5. **Centralized config:** every limit, the reserved-slug blocklist, and site constants live in `src/config/` — never hardcode them at call sites.
6. **Authorization on every mutation** (owner-or-admin), server-side validation always; client validation is UX only.

## Key product rules (quick reference — details in project-scope.md)

- **Slug, Display Name, and Theme are three independent fields (BR-1..BR-5d) — never conflate them:**
  - _Slug_: lowercase `a–z` only (no uppercase/digits/hyphens/spaces), 2–50 chars, globally unique, used exclusively to build the URL (`/tedx{slug}`), locked after first publication.
  - _Display Name_: any-case letters + spaces + accents + hyphens (no digits); NOT unique; freely editable anytime, even after publication; pre-filled at creation with a `TEDx`+capitalized-slug suggestion the user typically overwrites (e.g., `TEDxMcGill University`); shown in nav/page-title/Hero; the **only** field that's always required and can never be saved blank.
  - _Theme_: optional, ≤100 chars, short tagline/theme phrase shown as the Hero subtitle.
- **Publishing:** draft/snapshot model. Submit → completeness check → new snapshot + PublishRequest (one pending max, cancelable). Approve = atomic live-snapshot swap. Reject requires a reason. Owner may unpublish/republish freely; republishing an _unchanged_ approved snapshot needs no review. Admin may suspend/restore. Full-site review on every resubmission (no diffs in V1).
- **Sections:** Hero, About TED, About TEDx, disclaimer, footer always render. Only the Hero has blank-able organizer content — Theme and hero/background imagery — each falling back to a platform-provided default so the Hero never looks broken; About TED/About TEDx/disclaimer are static platform copy with no organizer-editable portion at all. Every other section (About, Venue, Speakers, Sponsors, Team, FAQ, Contact) auto-hides when empty, with **no** fallback content (sponsor tiers individually auto-hide too). **No Schedule section in V1** (deferred; will be an additive `EventContent` version bump).
- **Countdown** switches to "This event has taken place." after the event date.
- **Public homepage:** browsable without login; template cards show _Live Preview_ (opens `demoContent` through the real template, new tab, no auth) and _Edit_ (unauthenticated → login/signup → returns to event creation for that template). V1 shows one card (`aurora`), grid built to scale to future templates.
- **Limits (config/limits.ts):** 16 speakers, 30 team members, 30 sponsors, 30 FAQs, 10 MB/image.
- **Sponsor tiers:** PARTNER, PLATINUM, GOLD, SILVER, BRONZE, COMMUNITY.
- **Deletion:** confirmation required; soft delete if ever published, hard delete otherwise.
- **Preview links:** single active 256-bit token per event, revocable/regenerable, read-only, `noindex`.
- **Reports:** public form (category, explanation, optional email), honeypot + DB rate limit 3/IP/hour/site.
- **Emails (Resend):** verification, password reset, submission received, approved, rejected (with reason), suspended.

## Conventions

- TypeScript strict; no `any` escapes; domain errors as discriminated unions, not thrown strings.
- Zod schemas are the single source of validation truth, shared client and server.
- Editor mutations = Server Actions; public/unauthenticated endpoints = Route Handlers with rate limiting.
- Autosave: debounced (~1.5 s) per-section, status indicator (`Saving…/Saved/Save failed/Unsaved changes`) + manual "Save now"; last-write-wins with a warning on concurrent sessions.
- External user links: validate http/https, render with `target="_blank" rel="noopener noreferrer"`.
- Public sites: statically rendered with cache tags, revalidated on approve/unpublish/suspend; Lighthouse ≥ 90 target; `prefers-reduced-motion` respected.
- Definition of done for every task: typecheck + lint + relevant unit tests pass; authz on new mutations; limits from central config; loading/empty/error/success UI states; responsive + keyboard accessible.

## Out of scope for V1 (do not build)

Multiple templates, custom domains, i18n, analytics, AI, blogging, ticketing integrations, marketplace, public discovery, drag-and-drop editing, organizations/collaborators, schedule section, recorded talks, diff-based review. Architecture must keep these cheap to add later (e.g., `templateId` string column, `ownerId` as sole ownership link).
